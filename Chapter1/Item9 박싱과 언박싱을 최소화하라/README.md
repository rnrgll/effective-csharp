## 9. 박싱과 언박싱을 최소화하라

### 박싱과 언박싱

* `.NET`의 모든 타입은 `System.Object`를 상속받기 때문에, 값 타입(Value Type)을 참조 타입(Reference Type)으로 변환하거나 그 반대로 변환하는 것이 가능하다.

* 박싱 (Boxing)

  * 값 타입의 데이터를 참조 타입(`object` 등)으로 변환하는 과정

  * 스택(Stack)에 있는 데이터를 힙(Heap)에 새로 할당된 메모리로 복사하고, 이를 가리키는 포인터를 만든다.

* 언박싱 (Unboxing)

  * 박싱된 참조 타입의 객체에서 값 타입의 데이터를 다시 추출하는 과정

  * 힙에 있는 객체의 값을 스택으로 복사한다.

 

### 박싱과 언박싱의 단점

* 박싱과 언박싱은 단순한 할당보다 훨씬 무거운 작업이다.

* 이유

  * 메모리 할당\
    : 박싱 시 힙 영역에 새로운 메모리를 할당해야 합니다.

  * 가비지 컬렉션(GC)\
    : 힙에 할당된 박싱 객체는 결국 GC의 수거 대상이 되며, GC 빈도를 높여 프로그램 성능을 떨어뜨립니다.

  * 데이터 복사\
    : 값의 복사가 빈번하게 일어나 비용이 크다.

 

### 박싱/언박싱이 발생하는 상황

* 문자열 포맷팅 및 보간

  : `Console.WriteLine`이나 `String.Format` 등은 파라미터로 `object` 또는 `params object[]`를 받기 때문에, 값 타입의 데이터를 전달할 경우 박싱이 발생할 수 있다.

  ```csharp
  int i = 25;
  
  // ❌ 정수 i가 object로 박싱되어 전달됨
  Console.WriteLine($"A number: {i}"); 
  
  // ✅ 문자열은 참조 타입이므로 박싱 없이 전달됨
  Console.WriteLine($"A number: {i.ToString()}");
  
  ```

  ✅ 해결

  * 값 타입을 직접 전달하지 않고, `.ToString()`을 호출해 문자열(참조 타입)으로 변환한 뒤 전달한다.

  * 참고로 C# 9.0부터 동작 원리가 변경되어 더이상 박싱이 발생하지 않는다. 

* 비제너릭 컬렉션 사용

  : NET 1.x 시절의 `ArrayList`, `Hashtable` 등은 데이터를 `object`로 저장하므로, 값 타입을 넣을 때마다 무조건 박싱이 발생한다.

  ✅ 해결

  * .NET 2.0부터 도입된 `List<T>`, `Dictionary<TKey, TValue>` 등의 제네릭 컬렉션 사용

    ```csharp
    // ❌ 비제네릭 (박싱 O)
    ArrayList list = new ArrayList();
    list.Add(10); 
    
    // ✅ 제네릭 (박싱 X)
    List<int> list = new List<int>();
    list.Add(10);
    
    ```

* 인터페이스 형변환

  * 구조체(값 타입)가 인터페이스를 구현하고, 해당 인터페이스 타입으로 형변환될 때 박싱이 발생한다.
